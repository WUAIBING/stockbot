# TLFZ WorkBuddy 自动交易系统执行方案

## 配套文档

- 执行路线与验收标准：`WORKBUDDY_EXECUTION_PLAN.md`
- 每日成果与当前阶段：`WORKBUDDY_PROGRESS_LOG.md`

建议后续协作顺序：

1. 先读 `WORKBUDDY_PROGRESS_LOG.md`，确认当前做到哪一步。
2. 再读 `WORKBUDDY_EXECUTION_PLAN.md`，确认下一步按哪条路线推进。

## 1. 文档目的

本文档用于把当前 `TLFZ\workbuddy` 自动交易系统，从“已经能跑的自动化原型”，推进到“执行稳定、买卖协同、可持续进化、具备超越基准潜力”的专业化交易系统。

文档目标不是给出空泛方向，而是形成一套可以逐项执行、逐周复盘、逐阶段验收的行动方案，便于后续由我持续实施、由你配合验证与决策。

## 2. 当前系统状态判断

### 2.1 已完成基础能力

- 已完成 `scan -> decision -> buy -> smart-sell -> report` 自动链路。
- 已注册工作日自动调度任务，买入和报告链路已经能自动运行。
- 已完成 `trade` 接口限流初步治理，包括节流和 `112` 自动重试。
- 已完成交易后即时持仓同步，避免成交后账本长期漂移。
- 已完成半小时巡检式 `smart-sell` 调度。
- 已接入第一版持续进化模型，买入不再只是单因子排序。

### 2.2 仍然存在的核心问题

- 持仓账本尚未完全等于真实账户持仓全集，仍存在“账本纳管不全”的风险。
- 卖出逻辑当前仍以规则为主，模型在卖出侧参与度不足。
- 委托“长时间未成交/部分成交”的状态管理尚未完整。
- `report` 在接口空返回时仍需进一步保护，避免错误覆盖资产快照。
- 板块感当前使用行业热度代理，尚未形成更强的主线识别能力。

### 2.3 战略判断

当前系统已经从“无法依赖”进入“可运营，但未优化完成”的阶段。  
后续优化重点不应再是局部补丁式修复，而应进入体系化建设：

1. 先稳执行。
2. 再提买点质量。
3. 再提卖点灵活度。
4. 最后强化学习闭环。

## 3. 总体目标

### 3.1 核心目标

将 `TLFZ\workbuddy` 优化为一套具备以下特征的自动交易系统：

- 自动执行稳定，不出现魔幻委托、幽灵持仓、错误卖单。
- 买入纪律明确，能根据市场环境动态收放。
- 卖出具备盘中适应能力，能够结合趋势、盈利和环境灵活决策。
- 每日自动复盘，形成“决策 -> 成交 -> 结果 -> 学习 -> 下次修正”的闭环。

### 3.2 衡量标准

系统不是以“能运行”作为完成标准，而是以下指标逐步改善：

- 执行稳定性：
  - 计划任务按时触发率
  - 交易接口成功率
  - 账本与真实持仓一致率
- 交易质量：
  - 买入候选质量
  - 卖出及时性
  - 高盈利回吐控制
- 相对表现：
  - 胜率相对基准
  - 平均收益相对基准
  - 最大回撤相对基准

### 3.3 目标表述原则

不承诺“必然超越基准线”，但明确追求：

- 比当前系统更稳。
- 比当前系统更可解释。
- 比当前系统更有持续超额收益潜力。

## 4. 建设原则

### 4.1 先修底座，再谈收益

执行错误会直接吞噬模型收益，因此稳定性优先级高于策略花样。

### 4.2 规则与模型并用

不采用纯黑箱，也不只靠僵硬规则，而是：

- 规则负责底线和风控；
- 模型负责排序和动态偏好。

### 4.3 盘中判断与收盘学习分层

- 盘中系统要简洁、稳健、可执行；
- 收盘系统负责总结、归因、学习和参数刷新。

### 4.4 先解释，再优化

每一次买卖都要尽量能回答：

- 为什么买；
- 为什么卖；
- 为什么没买；
- 为什么没卖。

没有解释能力的优化，不进入正式系统。

## 5. 目标系统架构

### 5.1 执行层

负责自动化稳定运行：

- Windows 计划任务
- Phase runner
- 交易接口节流与重试
- 下单后即时持仓覆盖
- 收盘后全量对账

### 5.2 决策层

负责买卖判断：

- 买入：尾盘统一决策
- 卖出：盘中半小时巡检
- 评分：市场、行业、个股、分时

### 5.3 学习层

负责持续修正：

- 记录当天候选与最终成交
- 记录模式和环境下的真实表现
- 动态更新模型权重
- 输出学习结论到日报

### 5.4 观测层

负责复盘与排错：

- phase history
- latest phase status
- trade API log
- account summary
- NAV history
- model decisions

### 5.5 代码质量进化层

负责让“实现这套系统的代码能力”本身也进入持续进化，而不是停留在问题出现后再临时补丁。

- 输入来源：
  - 每日盘中巡检
  - 收盘复检
  - 异常执行样本
  - 策略与学习层的口径偏差样本
- 核心目标：
  - 提前暴露工程薄弱点
  - 把单次事故沉淀为长期防线
  - 持续降低同类错误复发概率
  - 减少代码实现对胜率和盈利率的拖累
- 主要输出：
  - 新增 fail-fast 校验
  - 新增最小回归测试
  - 新增巡检项与复检项
  - 新增代码实现规则与责任边界

代码质量进化层与其他四层的关系如下：

- 对执行层：
  - 重点识别任务漏跑、成功分支崩溃、pending 收口、账本漂移、source freshness 等工程问题。
- 对策略层：
  - 重点识别“策略意图是 A、代码实际执行成 B”的实现偏差。
- 对学习层：
  - 重点保证输入样本可信，防止 stale 数据、执行噪音、口径漂移污染学习。
- 对复检层：
  - 重点保证 review、summary、gate 结论真实反映执行事实，不给假绿灯。

- 进化闭环：
  - 巡检发现问题
  - 归类根因
  - 识别暴露出的代码能力短板
  - 新增防线
  - 用回归测试和次日巡检再次验证

- 根因分类固定分为：
  - 调度类
  - 执行类
  - 状态收口类
  - 时间语义类
  - freshness 类
  - 策略实现偏差类
  - 学习输入污染类
  - 复检失真类

## 6. 分阶段行动方案

## 6.1 第一阶段：V1 稳执行

### 目标

把系统从“可运行”升级到“可依赖”。

### 关键任务

1. 强化真实持仓覆盖
   - 每笔成交后即时对仓。
   - 收盘后全量持仓覆盖账本。
   - 所有真实持仓尽量纳入策略管理集合。

2. 完善订单状态管理
   - 增加 pending order 管理。
   - 区分 submitted / partial / filled / stale / rejected。
   - 防止重复报单。

3. 保护报告正确性
   - `balance/positions` 空返回时不覆盖旧资产快照。
   - 输出“接口异常”状态而不是写成零资产。

4. 继续优化交易稳定性
   - 进一步完善 `112` 限流退避。
   - 避免盘中短时间多笔委托相互挤压。

### 验收标准

- 不再出现“无仓卖出”“老底仓乱加仓”“幽灵 holding 乱卖”。
- `track_record` 与真实持仓明显更一致。
- `report` 不再出现零资产脏写。
- 委托失败可以通过日志快速定位具体原因。

---

## 6.2 第二阶段：V2 强买点

### 目标

提升买入质量，减少低质量交易。

### 关键任务

1. 固化多层评分买入
   - 市场评分
   - 行业热度代理评分
   - 个股趋势评分
   - 尾盘分时评分

2. 建立市场环境自适应
   - 活跃市：允许更积极进攻
   - 正常市：适中
   - 清淡市：自动收紧入场阈值

3. 建立低分过滤
   - 有信号不等于可买
   - 分数低于阈值直接不进入买单

4. 建立解释输出
   - 每只拟买标的要带上模型分数
   - 记录每个分量得分

### 验收标准

- 买入清单数量更可控。
- 弱市场日减少杂乱买单。
- 强市场日高分股优先更明显。
- 日报能解释为什么这些票被选中。

---

## 6.3 第三阶段：V3 灵活卖

### 目标

让卖出从“尾盘偏静态判断”升级为“盘中节奏化智能管理”。

### 当前已落地基础

- 每半小时智能卖出巡检：
  - `09:45`
  - `10:15`
  - `10:45`
  - `11:15`
  - `13:15`
  - `13:45`
  - `14:15`
  - `14:45`

### 下一步关键任务

1. 卖出决策从单规则升级为综合决策
   - 趋势衰减
   - 盈利状态
   - 持仓天数
   - 市场环境
   - 行业退潮状态

2. 建立盈利敏感度分层
   - 高盈利仓位：对滞涨/转弱更敏感
   - 中盈利仓位：趋势与利润并重
   - 低盈利或亏损仓位：更重防守逻辑

3. 建立卖出优先级
   - 若多只同时进入候选，先卖最弱或最该兑现的

4. 建立卖出巡检日志
   - 记录每次巡检：
     - 检查了哪些持仓
     - 为什么继续持有
     - 为什么卖出

### 验收标准

- 高盈利回吐问题下降。
- 转弱票的离场更及时。
- 卖出不再只靠收盘一次判断。
- 巡检时能够清晰解释“不卖”的理由。

---

## 6.4 第四阶段：V4 持续进化

### 目标

让系统不只是执行，而是持续积累经验并修正偏好。

### 关键任务

1. 完善模型学习样本
   - 候选记录
   - 实际成交记录
   - 平仓结果
   - 市场环境标签

2. 更新学习机制
   - 根据真实结果调整：
     - market 权重
     - sector 权重
     - stock 权重
     - flow 权重
   - 更新 mode bias 和 tier bias

3. 建立学习周期
   - 每日刷新轻量状态
   - 每周做一次人工复盘 + 结构调整

4. 输出学习报告
   - 哪类模式近期最有效
   - 哪类环境下容易失误
   - 哪些仓位配置需要收缩或放大

### 验收标准

- 报告中出现可读的学习结论，而不是静态数字堆砌。
- 权重调整与真实结果有因果解释。
- 一周后能够看见模型偏好开始稳定化。

### V4 硬约束

持续进化必须沿着“可验证、可回退、可冻结”的正向路径推进，不允许无约束漂移。

#### 1. 冻结基线

- 必须保留一份不随日常学习变化的稳定基线。
- 所有后续进化版本都要相对这份基线进行解释和比较。
- 没有基线，就无法判断系统是在进步还是在漂移。

#### 2. 样本门槛

- 不允许基于少量偶然交易直接调整权重。
- 至少达到最小样本门槛后，才允许进入自动微调。
- 样本不足时，系统只记录、不改参数。

#### 3. 冷却周期

- 不允许高频调参。
- 每次自动学习更新之间必须有冷却时间。
- 防止模型对短期噪声过度反应。

#### 4. 小步微调

- 权重、tier bias、mode bias 只允许在受限步长内变化。
- 一次调整不能跨越太大幅度。
- 即使方向错误，也应把风险控制在可回退范围内。

#### 5. 分层可学习

- 可学习项：
  - 市场权重
  - 行业权重
  - 个股权重
  - 分时权重
  - tier bias
  - mode bias
- 不可直接自动学习项：
  - 交易时窗
  - A 股交易规则
  - 仓位底线
  - 接口风控
  - 对仓与报告保护

#### 6. 变更留痕

- 每次学习更新都必须记录：
  - 为什么改
  - 用了哪些样本
  - 改了哪些参数
  - 改动前后差异
- 若因样本不足或冷却期跳过更新，也必须写入日志。

#### 7. 失败即冻结

- 一旦观察到：
  - 胜率恶化
  - 平均收益恶化
  - 回撤恶化
  - 高盈利回吐恶化
  - 执行错误上升
- 则优先冻结学习更新，回到稳定版，再复盘原因。

#### 8. 周期验收

- 日内只做记录和轻量校准。
- 真正的阶段判断以周为基本单位。
- 是否继续放开自动学习，必须以周级观察结果决定，而不是单日情绪决定。

## 7. 每周执行机制

## 7.1 日内执行节奏

### 上午

- `09:36`：加仓检查
- `09:45 / 10:15 / 10:45 / 11:15`：智能卖出巡检

### 下午

- `13:15 / 13:45 / 14:15 / 14:45`：智能卖出巡检
- `14:30`：预热扫描
- `14:49`：决策扫描
- `14:50`：买入 watch
- `15:06`：收盘报告

## 7.2 每日复盘重点

每天收盘后检查：

1. 自动化是否全部按时运行。
2. 当天买入候选和实际成交差异。
3. 卖出巡检是否有明确判断。
4. 委托是否出现失败、限流、未成交。
5. 账本、持仓、订单、报告是否一致。
6. 模型学习摘要是否有新变化。

### 每日巡检驱动的代码质量进化闭环

从现在开始，每天巡检不只服务“执行质量汇报”，还同时服务“代码质量进化”。

- 早盘巡检负责暴露：
  - 任务漏注册
  - phase 漏触发
  - 晨间数据缺失
  - source freshness 异常
- 盘中巡检负责暴露：
  - 成功分支崩溃
  - pending 残留
  - 账本与真实持仓不一致
  - summary 与执行事实不一致
- 收盘巡检负责暴露：
  - review 口径误判
  - learning sample 不可信
  - challenger / distill 接链失真
  - 学习放行结论与真实证据不一致

每日巡检结束后，固定输出一段“今日进化项”，最少包含：

- 今日暴露问题
- 根因分类
- 暴露出的代码能力短板
- 今日新增防线
- 今日新增验证
- 明日重点观察点

每次真实问题都必须至少沉淀为以下四项中的一项，优先做到多项同时落地：

- 一条 fail-fast 校验
- 一条最小回归测试
- 一条巡检清单项
- 一条复检摘要字段或 gate 规则

每周还要汇总一份“代码质量进化结果”，至少观察：

- 同类事故是否复发
- 新增防线是否真正生效
- 巡检能否更早发现问题
- 代码实现是否更少拖累策略胜率与盈利率

### 每日巡检固定 Checklist

从现在开始，每日巡检必须按固定 checklist 执行，不依赖临场记忆，不允许只凭印象汇报。

#### 早盘巡检 Checklist

- 调度与注册：
  - `09:31 opening-data`
  - `09:36 add-position`
  - `09:45 smart-sell`
  - challenger 对应早盘任务
  - 是否实际触发、是否落盘、是否有异常退出
- 晨间数据与 freshness：
  - `opening_tradability_latest.json` 是否为今日
  - `security_master` 是否更新
  - 候选池 `source_trade_date` 是否符合预期
- 执行层健康：
  - 是否出现任务漏跑
  - 是否出现 phase latest status 停在前一天
  - 是否出现静默失败
- 必出结论：
  - 早盘哪些节点正常
  - 哪些节点漏跑/错跑
  - 是否已影响主线或 challenger 执行

#### 盘中巡检 Checklist

- 买卖执行事实：
  - 是否真的报单
  - 是否真的成交
  - 成功分支是否完整收口
- 状态一致性：
  - `pending`
  - `positions`
  - `track_record`
  - `account_summary`
  - `latest_execution_result`
  - 是否互相一致
- 策略实现一致性：
  - 策略想做的动作是否真的被代码执行
  - 是否出现“应加仓未加”“应卖未卖”“大肉当普通票处理”
- 数据与时间语义：
  - 当次 phase 使用的数据是否 fresh
  - `trade_date / generated_at / source_trade_date` 是否口径一致
- 必出结论：
  - 本轮执行质量如何
  - 是否存在执行层拖累策略收益的代码问题
  - 是否需要立刻修复

#### 收盘巡检 Checklist

- 复检层健康：
  - `close-node` 是否完整执行
  - `review` / `gate` / `summary` 是否真实反映执行事实
  - 是否存在误报 stale、误给 ok、漏记 blocker
- 学习层输入质量：
  - 今日样本是否可信
  - 是否存在 stale source、执行噪音、账本污染
  - 是否适合进入学习候选
- challenger / distill 接链：
  - challenger 今日是否真正使用预期候选池
  - distill / route / shadow ledger 是否正常刷新
- 必出结论：
  - 今日复检是否通过
  - 哪些异常只是观察项
  - 哪些异常会影响下一交易日

#### 代码质量进化 Checklist

## 7.4 划线与对标计划

### 目标

建立一套“能真实反映系统改进是否有效”的曲线体系，用于区分：

- 市场本身在帮忙；
- 执行层是否更稳定；
- 策略质量是否提升；
- 模型学习是否沿着正向路径推进。

### 基本原则

- 不只画一条收益线，避免把市场行情、执行修复和模型进化混为一谈。
- 所有曲线都必须能追溯到明确数据源，不允许靠主观估计补值。
- 结果线、执行线、策略线、学习线分层观察，避免单指标误判。
- 当前跑什么交易宇宙，就用什么主基准；不把单一基准永久固定。

### 第一层：结果曲线

用于回答“账户结果是否真的在变好”。

- 总资产曲线：
  - 数据源：`v10_nav_history.csv`
  - 用途：看账户净值总体变化和回撤。
- 累计已实现盈亏曲线：
  - 数据源：`v10_nav_history.csv` + `v10_track_record.csv`
  - 用途：看系统把多少成果从纸面兑现成落袋结果。
- 浮动盈亏曲线：
  - 数据源：`v10_account_summary_latest.json`
  - 用途：区分“已兑现改善”和“仍停留在持仓账面上的改善”。

### 第二层：执行曲线

用于回答“系统是不是越来越稳，而不是只在好运时赚钱”。

- 计划任务准点完成率：
  - 数据源：`automation_status/phase_history.csv`
  - 用途：看自动化是否按时收尾，而不是只启动不结束。
- 买单成功率 / 卖单成功率：
  - 数据源：`v10_trade_api_log.jsonl`
  - 用途：衡量交易链路真实可执行性。
- `112` 限流次数：
  - 数据源：`v10_trade_api_log.jsonl`
  - 用途：评估节流与退避是否真的起效。
- 废单率 / 重复报单次数：
  - 数据源：`v10_trade_api_log.jsonl` + `v10_pending_orders.json`
  - 用途：识别执行层是否仍吞噬策略收益。

### 第三层：策略曲线

用于回答“决策本身有没有在进步”。

- 本周新增平仓收益曲线：
  - 数据源：`v10_track_record.csv`
  - 用途：剔除历史修正后，观察本周新交易的真实兑现质量。
- 胜率与平均单笔收益曲线：
  - 数据源：`v10_track_record.csv`
  - 用途：观察策略结果是否稳定，而不是靠少数大赚单支撑。
- 分层表现曲线：
  - 数据源：`v10_track_record.csv` + `v10_model_decisions.jsonl`
  - 用途：按 `tier`、`mode`、市场环境拆解表现，找出有效模式和失效模式。
- 午间复盘命中率曲线：
  - 数据源：`v10_midday_review_latest.json` + 后续成交记录
  - 用途：验证中场校准是否真的提升下午判断准确率。

### 第四层：学习曲线

用于回答“模型是不是在受约束地进化，而不是漂移”。

- 样本积累曲线：
  - 数据源：`v10_evolving_model_state.json`
  - 用途：看是否达到足够学习样本。
- 权重变化曲线：
  - 数据源：`v10_evolving_model_state.json` + `v10_evolving_model_changelog.jsonl`
  - 用途：看 `market/sector/stock/flow` 权重是如何变化的。
- bias 变化曲线：
  - 数据源：`v10_evolving_model_state.json` + `v10_evolving_model_changelog.jsonl`
  - 用途：看 `tier_bias` 和 `mode_bias` 是否小步、可解释地调整。
- 学习状态曲线：
  - 数据源：`v10_evolving_model_changelog.jsonl`
  - 用途：统计 `apply_update`、`skip_update`、`guardrail_cooldown`、`insufficient_samples` 等状态，验证 guardrails 是否真正工作。

### 动态基准设计

- 主基准：
  - 与当前真实交易宇宙一致。
  - 当前阶段默认使用 `中证1000`。
- 辅基准：
  - 用于观察成长风格和市场温度。
  - 当前阶段建议同时观察 `创业板指`。
- 后续切换规则：
  - 若交易宇宙调整到 `创业板`、`主板`、`北交所`、`ETF` 或混合池，则主基准同步切换；
  - 不允许“交易池已经变了，基准仍停留在旧宇宙”。

### 图上必须标注的关键事件

为了把“改进动作”和“结果变化”建立更真实的对应关系，所有关键曲线图都要同步标记版本节点：

- 半小时 `smart-sell` 上线；
- 即时对仓与收盘全量覆盖上线；
- `112` 限流退避参数调整；
- 午间复盘 `midday-review` 上线；
- 卖单挂单占用扣减上线；
- `buy-watch` 去掉重复扫描并加入超时保护；
- 后续任何主基准切换、交易宇宙切换、重要模型权重更新。

### 第一版先落地哪些线

第一版不追求大而全，先落地最有判断价值的五条线：

1. 总资产曲线
2. 累计已实现盈亏曲线
3. 本周新增平仓收益曲线
4. 买卖成功率与 `112` 次数曲线
5. 主基准与辅基准对比曲线

### 第二版再扩展哪些线

等样本更稳定后，再扩展：

1. `tier` / `mode` 分层收益曲线
2. 午间复盘命中率曲线
3. 模型权重与 bias 变化曲线

### 落盘格式与目录规范

- 统一输出目录：
  - `a-share-analyst/curve_observatory/`
- 数据目录：
  - `a-share-analyst/curve_observatory/data/`
  - 主格式使用 `CSV`
  - 作用是保存可重复计算、可追溯、可被脚本重新出图的原始时序数据
- 静态图目录：
  - `a-share-analyst/curve_observatory/charts/png/`
  - 主格式使用 `PNG`
  - 作用是用于复盘、归档、周报和直接查看
- 交互图目录：
  - `a-share-analyst/curve_observatory/charts/html/`
  - 主格式使用 `HTML`
  - 作用是用于放大、悬停查看和临时分析，不替代 `PNG` 归档
- 解读目录：
  - `a-share-analyst/curve_observatory/reports/`
  - 主格式使用 `MD`
  - 作用是记录口径、结论、关键版本节点和周度解释

### 命名规范

- 数据文件：
  - `curve_<metric_name>_<granularity>.csv`
  - 例如：
    - `curve_nav_daily.csv`
    - `curve_realized_pnl_daily.csv`
    - `curve_trade_success_weekly.csv`
- 静态图：
  - `curve_<theme>_<window>_<yyyymmdd>.png`
  - 例如：
    - `curve_results_weekly_20260618.png`
    - `curve_execution_daily_20260618.png`
- 交互图：
  - `curve_<theme>_<window>_<yyyymmdd>.html`
  - 例如：
    - `curve_results_weekly_20260618.html`
- 解读文档：
  - `curve_review_<yyyy_ww>.md`
  - 例如：
    - `curve_review_2026_w25.md`

### 格式选择原则

- `CSV` 是主数据格式，不使用 `XLSX` 作为底层主存储。
- `PNG` 是正式归档图格式，保证任何时候都能直接打开查看。
- `HTML` 是增强分析格式，用于交互而不是归档唯一来源。
- `MD` 只保存解释，不承载主时序数据。
- `XLSX` 如需导出，只作为临时汇报或人工查看副本，不作为标准源数据。

### 验收标准

- 任何一条曲线都能明确说出“它代表什么，不代表什么”。
- 看到收益变化时，能够分辨是市场推动、执行改善、还是策略提升。
- 看到模型参数变化时，能够对应到样本积累和变更日志，而不是黑箱漂移。
- 每周复盘可以基于这些曲线做出“继续放开 / 暂停优化 / 回退稳定版”的决策。

## 8. 是否能超越基准线

### 8.1 结论表述

不承诺必然超越基准线，但本方案的设计目标是：

- 先提升执行质量；
- 再提升买卖决策质量；
- 最后提升相对收益表现。

### 8.2 能形成超额收益的逻辑来源

如果系统最终能跑赢基准，其来源不会是单点神招，而是以下组合：

1. 更少的执行损耗。
2. 更少的低质量买入。
3. 更及时的卖出兑现。
4. 更适应市场风格变化的动态权重。

### 8.3 风险提示

以下情况会削弱跑赢概率：

- 市场风格突变；
- 主线判断失效；
- 板块代理不够准确；
- 接口层不稳定；
- 样本期过短造成学习漂移。

因此，超越基准只能通过持续观察和迭代逼近，不能靠承诺得到。
